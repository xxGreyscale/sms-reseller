<objective>
Add the wallet-service side of the credit lifecycle (D-15): per-lot allocation in the
reservation result so messaging can map each recipient to a specific credit lot; the
`consumeFromLot` and `releaseFromLot` LotService operations; and a `MessagingEventConsumer`
that idempotently applies CONSUME / RELEASE / REFUND when messaging-service publishes
`MessageAccepted` / `MessageReleased` / `MessageRefundDue` on `messaging.events`.

CRITICAL GAP FOUND IN PHASE 3: `ReservationResult.lotIds` is a list of DISTINCT touched lots
only — it does NOT expose how many credits came from each lot (ReservationService.reserve line
74 adds each lot id once, with no per-lot count). D-13 requires per-message lot assignment, so
this plan MUST enrich the reservation result with per-lot counts WITHOUT breaking existing
callers.

Purpose: wallet is the single owner of the ledger; messaging emits events, wallet acts.
Output: per-lot allocation contract + idempotent consume/release/refund consumer.
</objective>

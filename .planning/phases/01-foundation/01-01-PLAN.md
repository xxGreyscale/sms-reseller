<objective>
Establish the Gradle multi-module monorepo skeleton that every other Phase 1 plan and every
future phase builds on: root build + version catalog + buildSrc convention plugins, all 8 empty
service modules, both shared library modules, and the Wave 0 placeholder test stubs that give the
project a green test baseline.

Purpose: Without this skeleton nothing compiles. The convention plugins (`spring-boot-service`,
`spring-boot-library`) are the single source of truth for Java 21 toolchain, annotation-processor
order, and the bootJar/plain-jar split — they prevent the "bootJar fails on library module" and
"Java version mismatch" pitfalls documented in RESEARCH.md.
Output: A repository where `./gradlew build` compiles all 10 modules and runs the Wave 0 tests green.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| developer machine → Gradle build | Build scripts and buildSrc plugins execute with developer privileges |
| Gradle build → Maven Central / Gradle Plugin Portal | External dependency resolution crosses the network boundary |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Tampering | Dependency resolution (Maven Central) | mitigate | Pin all versions in `gradle/libs.versions.toml`; Boot BOM manages transitive versions deterministically; no dynamic `+` version ranges |
| T-01-02 | Tampering | Gradle wrapper distribution | mitigate | Wrapper pinned to gradle-8.11.1; distributionUrl points at the canonical services.gradle.org URL |
| T-01-03 | Elevation of Privilege | buildSrc convention plugin code | accept | buildSrc runs trusted first-party build logic only; no third-party plugin code beyond Spring Boot + dependency-management which are pinned |
| T-01-SC | Tampering | Gradle/npm/Maven installs | mitigate | All packages in 01-RESEARCH.md Package Legitimacy Audit are Maven Central / npm official and Approved; no [ASSUMED]/[SUS] application packages in this plan (Docker images are handled in Plan 03). No package install checkpoint required here |
</threat_model>

# Quick Task 260625-p5c: Azampay `sandbox` profile

## Goal

Let the **real** Azampay sandbox flow be exercised end-to-end without production
merchant onboarding or the (still-unwritten) HMAC validator — by introducing a
`sandbox` Spring profile that activates the real gateway + token provider while
falling back to the always-true stub signature validator.

## Why this is safe

Under `sandbox`, signature validation is the always-true stub, but the callback path
keeps its defense-in-depth: `CallbackController` rejects any `utilityRef` not present
in the `payments` table, and wallet crediting is idempotent (`INSERT … ON CONFLICT DO
NOTHING`). A forged/duplicate sandbox callback therefore cannot mis-credit. The `prod`
profile stays correctly blocked (no `WebhookSignatureValidator` bean) until a real
`HmacSignatureValidator(@Profile("prod"))` is written.

## Bean wiring after change (profile = `sandbox`)

| Bean | Profile before | Profile after | Active under sandbox? |
|------|----------------|---------------|-----------------------|
| AzampayPaymentGateway | `prod` | `{prod, sandbox}` | ✅ real gateway |
| AzampayTokenProvider | `prod` | `{prod, sandbox}` | ✅ |
| StubSignatureValidator | `stub` | `{stub, sandbox}` | ✅ permissive validator |
| StubPaymentGateway | `stub` | `stub` (unchanged) | ❌ (not pulled in) |

No duplicate `PaymentGateway` / `WebhookSignatureValidator` beans under any single
profile. `sandbox` is standalone (not in the `dev→stub` / `staging→stub` groups).
